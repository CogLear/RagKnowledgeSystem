# 保险系统架构设计

## 一、架构设计原则

互联网保险系统采用云原生微服务架构，遵循以下核心设计原则：

| 原则 | 说明 |
| --- | --- |
| 高可用 | 多活部署，故障自动切换，SLA ≥ 99.95% |
| 可扩展 | 水平扩容，按需扩展，避免单点瓶颈 |
| 松耦合 | 服务间通过 API/消息通信，独立部署和演进 |
| 可观测 | 全链路监控，日志、链路追踪、指标一体化 |
| 安全合规 | 数据加密，权限分级，满足保险行业监管要求 |

### 2.2 核心中间件选型

| 中间件 | 用途 | 选型理由 |
| --- | --- | --- |
| 注册中心 | 服务发现与配置管理 | Nacos / Apollo |
| 消息队列 | 异步解耦，事件驱动 | Kafka |
| 缓存 | 热数据访问加速 | Redis Cluster |
| 熔断限流 | 流量保护 | Sentinel |
| 分布式事务 | 跨服务一致性 | Seata AT 模式 |
| 文件存储 | 合同、发票、影像存储 | MinIO（S3 兼容） |
| 数据库 | 主数据存储 | MySQL 主从 + 读写分离 |
| 搜索引擎 | 全文检索、日志分析 | Elasticsearch |
| 向量数据库 | 语义检索、智能问答 | Milvus |

## 三、服务拆分设计

### 3.1 服务划分维度

按业务边界（Domain-Driven Design）划分服务，每个服务对应一个独立的业务域：

| 服务名称 | 职责 | 核心技术栈 |
| --- | --- | --- |
| product-service | 产品域：产品配置、费率管理 | Spring Boot + MySQL |
| policy-service | 保单域：保单生命周期管理 | Spring Boot + MySQL + Redis |
| underwriting-service | 承保域：核保、风控 | Spring Boot + MySQL + 规则引擎 |
| claim-service | 理赔域：理赔全流程处理 | Spring Boot + MySQL + Kafka |
| policy-holder-service | 客户域：投保人/被保人管理 | Spring Boot + MySQL + ES |
| payment-service | 支付域：保费收取、理赔出款 | Spring Boot + MySQL + 支付网关 |
| commission-service | 佣金域：代理人/渠道佣金计算与发放 | Spring Boot + MySQL |
| customer-service | 客服域：客服工单、智能问答 | Spring Boot + MySQL + Milvus |
| data-service | 数据域：报表、BI、数据分析 | Spring Boot + MySQL + ES |

### 3.2 服务间通信模式

| 通信模式 | 适用场景 | 技术选型 |
| --- | --- | --- |
| 同步 REST | 需要实时响应的场景 | OpenFeign + Nacos |
| 异步消息 | 跨服务事件通知、最终一致性 | Kafka |
| 事件驱动 | 状态变更触发后续流程 | Kafka + 事件溯源 |

**示例：投保成功触发后续流程**

```
投保申请 → 核保通过 → 发送「保单生效事件」到 Kafka
                     ↓
         ┌─────────────┼─────────────┐
         ↓             ↓             ↓
    支付服务监听    短信服务监听   佣金服务监听
    (触发收款)     (发送通知)    (记录佣金)
```

## 四、数据库设计

### 4.1 分库分表策略

核心业务表按业务域拆分到不同数据库：

```
┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
│   policy_db      │  │   claim_db       │  │   product_db     │
│   (保单库)        │  │   (理赔库)        │  │   (产品库)        │
├──────────────────┤  ├──────────────────┤  ├──────────────────┤
│  policy_info     │  │  claim_record    │  │  product_info    │
│  policy_holder   │  │  claim_attachment │  │  product_rate    │
│  policy_beneficiary│ │  claim_audit    │  │  insurance_clause │
└──────────────────┘  └──────────────────┘  └──────────────────┘
```

### 4.2 读写分离

- **读库**：多个从库承载查询流量（报表、详情页）
- **写库**：主库承担写入，遵循"主库优先"原则
- **缓存**：热点数据（产品信息、保单状态）加载至 Redis

## 五、高可用与容灾设计

### 5.1 多活部署

```
         用户请求
             │
     ┌───────▼───────┐
     │   DNS 智能调度  │
     └───────┬───────┘
        ┌────┴────┐
        ↓         ↓
   [主集群]    [备集群]
   (A机房)    (B机房)
   Active     Standby
```

- **同城双活**：主备机房部署，服务级别故障自动切换
- **单元化部署**：核心服务按用户维度隔离，单机房故障不影响其他用户

### 5.2 熔断降级

使用 Sentinel 实现服务熔断和降级：

| 策略 | 触发条件 | 处置方式 |
| --- | --- | --- |
| 熔断 | 某服务 N 次失败率超阈值 | 快速返回降级响应，不再调用 |
| 限流 | QPS 超过阈值 | 拒绝超出流量，返回"系统繁忙" |
| 冷启动 | 新服务刚启动 | 逐步放行，避免冷库冲击 |
| 均值负载 | 响应时间超时 | 走降级逻辑，返回缓存或默认值 |

## 六、安全架构

### 6.1 数据安全

| 措施 | 说明 |
| --- | --- |
| 传输加密 | 全站 TLS 1.2+，敏感接口强制 HTTPS |
| 存储加密 | 敏感字段（身份证、银行卡）AES-256 加密存储 |
| 脱敏展示 | 日志和展示页面中敏感数据脱敏 |
| 备份加密 | 备份文件加密存储，异地容灾 |

### 6.2 访问控制

- **接口鉴权**：JWT Token + 签名校验
- **数据权限**：基于 RBAC + ABAC 的行级/字段级权限
- **操作审计**：所有敏感操作写入审计日志，不可篡改

### 6.3 合规要求

| 要求 | 实现方式 |
| --- | --- |
| 等保三级 | 安全审计、访问控制、入侵防范等措施落地 |
| 个人信息保护 | 隐私合规评审、数据分类分级、最小化采集 |
| 保险监管报送 | 定期生成监管报送数据，满足监管格式要求 |

## 七、关键设计决策

### 决策一：为何选择 Kafka 而不是 RabbitMQ？

- **吞吐能力**：Kafka 吞吐量可达百万级 QPS，满足保险大促场景
- **消息回溯**：支持按 offset 回溯，利于故障排查和重试
- **生态完善**：与 Flink、ES 等系统天然集成，利于数据分析

### 决策二：为何选择 Seata 而非 Saga？

- AT 模式对业务代码零侵入，开发成本低
- 补偿逻辑由框架自动处理，减少人工错误
- 适合保险业务中大量存在的"短事务"场景

### 决策三：为何引入 Milvus 向量数据库？

- 支持语义检索，实现"智能问答"功能
- 支持相似文档检索，提升知识库问答准确率
- 与现有 MySQL + ES 体系互补，不重复建设
