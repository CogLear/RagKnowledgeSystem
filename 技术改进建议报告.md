## 一、项目概述

**RagKnowledgeSystem** 是一个企业级 RAG（检索增强生成）知识库系统，采用 Spring Boot + React 多模块架构。

### 技术栈

| 层级 | 技术 | 版本 |
|------|------|------|
| 后端框架 | Spring Boot | 3.5.13 |
| ORM | MyBatis-Plus | 3.5.14 |
| 认证 | Sa-Token | 1.43.0 |
| 缓存/分布式锁 | Redisson | 4.0.0 |
| 向量数据库 | Milvus SDK | 2.6.6 |
| 文档解析 | Apache Tika | 3.2.3 |
| 前端框架 | React + TypeScript | 18 / 5.5 |
| 构建工具 | Vite | 5.4 |
| 状态管理 | Zustand | - |

### ✅ 2.2 批量创建 Chunk 索引分配竞态条件

**问题位置**: `KnowledgeChunkServiceImpl.batchCreate()` / `create()`

**修复方案**: 使用 Redisson 分布式锁，对每个 `docId` 单独加锁，确保索引查询-分配的原子性。

```java
String lockKey = CHUNK_INDEX_LOCK_PREFIX + docId;
RLock lock = redissonClient.getLock(lockKey);
try {
    lock.lock();
    // 查询-分配-插入在锁内执行
    doBatchCreateWithLock(...);
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

**修复文件**:
- `bootstrap/.../knowledge/service/impl/KnowledgeChunkServiceImpl.java` - 新增分布式锁

### ✅ 2.3 Milvus 写入失败数据不一致

**问题位置**: `KnowledgeChunkServiceImpl.create()` / `batchCreate()`

**修复方案**: 改为**先写 Milvus，成功后再写 MySQL**。如果 Milvus 失败，MySQL 不会执行；如果 Milvus 成功但 MySQL 失败，事务回滚。

```
修复前：MySQL插入 → Milvus同步（失败时数据不一致）
修复后：Milvus写入 → MySQL插入（失败时无数据不一致）
```

**修复文件**:
- `bootstrap/.../knowledge/service/impl/KnowledgeChunkServiceImpl.java` - 调整写入顺序

---

### ✅ 2.4 前端状态管理耦合

**问题位置**: `frontend/src/stores/authStore.ts`

**修复方案**: 引入事件机制解耦，`authStore` 发出登录/登出事件，`chatStore` 监听并响应。

```typescript
// authEvents.ts - 事件发射器
class AuthEventEmitter {
  private listeners: Set<AuthEventListener> = new Set();
  subscribe(listener: AuthEventListener): () => void { ... }
  emit(type: "login" | "logout") { ... }
}
export const authEvents = new AuthEventEmitter();

// authStore.ts - 发出事件
authEvents.emit("login");

// chatStore.ts - 监听事件
authEvents.subscribe((event) => {
  if (event.type === "login" || event.type === "logout") {
    useChatStore.setState({ /* 重置状态 */ });
  }
});
```

**修复文件**:
- `frontend/src/stores/authEvents.ts` - 新增事件发射器
- `frontend/src/stores/authStore.ts` - 移除对 chatStore 的直接依赖
- `frontend/src/stores/chatStore.ts` - 添加事件监听

## 三、高优先级问题（待修复）

### 3.1 Chunk ID 不一致

**问题位置**: `KnowledgeChunkServiceImpl.doRebuildByDocId()`

**现象**: 重建向量索引时使用新雪花 ID 作为 `chunkId`，但 MySQL 中 chunk 的 ID 是自增 Long，导致 Milvus 中的 `doc_id` 与 MySQL 中的 `id` 不一致。

```java
.chunkId(IdUtil.getSnowflakeNextIdStr())  // ❌ 使用新雪花 ID
// 应该是 chunkMapper 插入后返回的 id
```

**建议**: 重建索引时应使用 MySQL 中实际的 chunk ID，而非生成新 ID。

---

## 四、中优先级问题（待修复）

### 4.1 意图树缓存无本地层

**问题位置**: `IntentTreeCacheManager`

**现象**: 意图树每次从 Redis 加载，无本地缓存，高并发时对 Redis 压力较大。

**建议**: 实现 L1本地缓存(L1 Caffeine) + L2 Redis缓存 + L3 DB 的三级缓存架构。

```java
// L1: 本地缓存，容量 1000，过期 5 分钟
LoadingCache<String, IntentTree> localCache = Caffeine.newBuilder()
    .maximumSize(1000)
    .expireAfterWrite(Duration.ofMinutes(5))
    .build();

// L2: Redis 缓存，现有实现
// L3: DB 加载
```

---

### 4.2 意图分类串行执行

**问题位置**: `DefaultIntentClassifier.classifyTargets()`

**现象**: 所有叶子节点一次性发给 LLM 分类，意图多时 token 消耗大、延迟高。

**建议**: 实现两阶段分类:
1. 第一阶段: 先按 Domain 分类，确定问题领域
2. 第二阶段: 仅对该 Domain 下的 Topic 并行分类

---

### 4.3 向量维度硬编码

**问题位置**: `MilvusVectorStoreService` 多处

**现象**: 向量维度硬编码为 4096，不支持其他维度的 embedding 模型。

```java
final int dim = 4096;  // ❌ 硬编码
```

**建议**: 从 `t_knowledge_base.embedding_model` 动态获取维度，或在创建 Collection 时从配置读取。

---

### 4.4 Provider 硬编码

**问题位置**: `RoutingLLMService`

**现象**: 使用字符串比较获取 Provider，新增 Provider 需要修改代码。

```java
clientsByProvider.get(target.candidate().getProvider())  // ❌ 字符串比较
```

**建议**: 使用枚举或配置驱动，通过策略模式 + 工厂模式实现插件化 Provider。

---

### 4.5 检索结果无超时控制

**问题位置**: `MultiChannelRetrievalEngine`

**现象**: 多 Collection 并行检索无超时控制，某慢 Collection 会阻塞整体响应。

**建议**:
```java
// 使用 CompletableFuture 超时控制
List<RetrievalResult> results = CompletableFuture.allOf(
    channels.stream()
        .map(c -> CompletableFuture.supplyAsync(() -> c.retrieve())
            .orTimeout(3, TimeUnit.SECONDS)
            .exceptionally(ex -> null))
        .toArray(CompletableFuture[]::new)
).get(5, TimeUnit.SECONDS);
```

---

### 4.6 S3 Bucket 错误处理不当

**问题位置**: `KnowledgeBaseServiceImpl.create()`

**现象**: `BucketAlreadyOwnedByYouException` 也抛出异常，但本账户已创建实际上应该视为成功。

**建议**: `BucketAlreadyOwnedByYouException` 应视为成功，直接返回。

---

### 4.7 ChatStore 职责过重

**问题位置**: `frontend/src/stores/chatStore.ts`

**现象**: chatStore 承担了会话管理、消息流、反馈、标题更新等太多职责。

**建议**: 拆分为多个 Store:
- `sessionStore`: 会话列表管理
- `messageStore`: 消息管理
- `streamingStore`: 流式状态管理

---

## 五、低优先级问题（待修复）

### 5.1 Prompt 模板硬编码

**问题位置**: `RAGConstant.java`

**现象**: Prompt 模板路径硬编码在 classpath，不支持动态修改。

**建议**: 支持从数据库或配置中心加载 Prompt 模板，实现版本管理和灰度能力。

---

### 5.2 FixedSizeTextChunker 边界调整

**问题位置**: `FixedSizeTextChunker.adjustToBoundary()`

**现象**: 边界调整可能落在单词中间，切碎词汇影响检索质量。

**建议**: 增加单词边界检测，如果 targetEnd 刚好在单词中间则回退到上一个完整单词。

---

### 5.3 MCP 工具调用无重试

**问题位置**: `MCPToolExecutor`

**现象**: 工具调用失败无重试机制，网络抖动时直接失败。

**建议**: 实现带指数退避的重试机制:
```java
@Retry(maxAttempts = 3, delay = 1000, multiplier = 2)
public MCPResponse execute(MCPRequest request) {
    // ...
}
```

---

### 5.4 首包探测超时较长

**问题位置**: `RoutingLLMService.streamChat()`

**现象**: `FIRST_PACKET_TIMEOUT_SECONDS = 60` 秒较长，可能导致差体验。

**建议**: 降低到 30 秒，并在前端显示更明确的加载状态。

---

### 5.5 摘要生成无质量评估

**问题位置**: `MySQLConversationMemorySummaryService`

**现象**: 摘要生成后无质量评估，过低质量的摘要会影响后续对话。

**建议**: 增加摘要质量评分，低分时重新生成或保留更多原始消息。

---

### 5.6 Milvus Filter 字符串拼接

**问题位置**: `MilvusVectorStoreService.deleteDocumentVectors()`

**现象**: 使用字符串拼接构建 filter，存在潜在注入风险。

```java
String filter = "metadata[\"kb_id\"] == \"" + kbId + "\"";  // ⚠️ 字符串拼接
```

**建议**: 虽然 kbId 来自内部系统，但仍建议使用 Milvus SDK 的参数化查询接口。

---

### 5.7 前后端错误码不同步

**问题位置**: `frontend/src/services/api.ts`

**现象**: 后端返回 `{ code: "A000001", message: "..." }`，前端直接作为 `Error.message` 抛出，丢失了 `code` 字段信息。

**建议**:
1. 维护一份共享的错误码枚举（可考虑单独 npm 包）
2. 在前端构建错误码到错误类型的映射
3. 避免重复 toast 提示
